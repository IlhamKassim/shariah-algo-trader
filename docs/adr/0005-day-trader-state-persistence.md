# Local state persistence for the Day Trader, reconciled against Alpaca on startup

Once the Day Trader gained an all-day Opening Range Breakout scanner (`day_trader/jobs/intraday_scan.py`) alongside the 9:31 AM Gap and Go entry, positions can now stay open for hours instead of minutes, which made an existing gap worth closing: `DayTraderState` (`day_trader/state.py`) was a bare in-process dataclass with zero durability — a process restart produced a brand-new empty state, silently losing every open position's stop-loss level, trailing-stop high-water mark, and profit-target basis. None of that is recoverable from Alpaca, which has no field for our own derived risk parameters. We added `day_trader/state_persistence.py`, which writes a JSON snapshot of `positions`, `traded_today`, `starting_equity`, and `circuit_broken` to a local file at the end of every state-mutating job, and a `reconcile_on_startup()` step (called once in `main()` before the scheduler starts) that reloads it and cross-checks every symbol against Alpaca's live `/v2/positions` before trusting it.

This deliberately does not contradict `docs/adr/0002-alpaca-as-portfolio-state-source-of-truth.md`, which established Alpaca as the sole authority for portfolio state to avoid local/remote divergence. Alpaca remains that authority here too: a cached position is only restored if Alpaca itself confirms the symbol is still open with a matching quantity; any live Alpaca position with no matching cached record is treated as an orphan with no recoverable risk parameters and is liquidated immediately rather than held un-managed or guessed at. The local file only ever caches *our own strategy's derived numbers* (stop level, trailing peak, target basis) — data Alpaca has no concept of and cannot answer for on its own — so there is no new "what do we hold" duplication of the kind `0002` was written to prevent, only a cache of numbers Alpaca was never going to have anyway.

The alternative was external storage (a hosted Postgres/Redis instance, or a Render persistent disk add-on) which would additionally survive a full redeploy, not just a same-container crash — this local JSON file only protects against the latter, since a genuine redeploy starts from a fresh container with a fresh disk. That gap is accepted for now rather than solved: it mirrors the Tier 2 float-data-provider decision in `docs/adr/0004-day-trader-aggressive-retune.md` — a real infrastructure investment for what is explicitly a benchmark strategy, not the project's core deliverable. Revisit if restart frequency in production turns out to matter more than expected.
